## 1. Idempotency Strategy

### Definition

Every state-changing operation must be safely re-executable without side effects.

### Idempotency Key

**For webhook-triggered operations (inbound messages, delivery updates):**

```
idempotency_key = hash(workspace_id + channel_type + channel_message_id + operation_type)
```

**For API operations (agent actions):**

```
idempotency_key = request_id (provided by frontend or generated by backend)
```

### Storage

Create table:
```sql
CREATE TABLE idempotency_keys (
  id UUID PRIMARY KEY,
  workspace_id UUID NOT NULL,
  idempotency_key TEXT NOT NULL,
  operation_type TEXT,
  request_body JSONB,
  response_body JSONB,
  status INT,
  created_at TIMESTAMPTZ,
  
  UNIQUE(workspace_id, idempotency_key)
);
```

### Implementation Rules

1. **On incoming request/webhook:**
   - Compute idempotency_key
   - Query `idempotency_keys` for matching key within workspace
   - If exists (status = 200): return cached response immediately; do NOT re-execute
   - If exists (status != 200): optionally retry; otherwise return cached error
   - If not exists: proceed with operation

2. **After successful operation:**
   - Store request + response in `idempotency_keys` with status = 200
   - If operation fails: store status = error_code (e.g., 502)

3. **Lifetime:**
   - Keep idempotency records for 30 days
   - Older records can be archived or deleted

### Example: Inbound WhatsApp Message

```
1. Webhook received: { workspace_id: X, channel_message_id: "123abc", content: "..." }
2. Compute: idempotency_key = hash("X:whatsapp:123abc:inbound_message")
3. Query: SELECT * FROM idempotency_keys WHERE workspace_id = X AND idempotency_key = '...'
4. If found (200): return cached conversation_id; do NOT create duplicate message
5. If not found: 
   a. Create contact (if new)
   b. Create/find conversation
   c. Create message record
   d. Store idempotency record (success)
   e. Broadcast realtime
```

**Outcome:** Receiving the same webhook 10 times creates only 1 message record.

## 2. Webhook Replay Handling

### Scenario

Evolution API retries failed webhook delivery. Same webhook payload arrives 2–3 times.
Same for Meta, SendGrid, or future integrations.

### Strategy

**All webhooks are idempotent by design (see Section 1).**

No special "replay detection" logic needed. The idempotency key mechanism handles this.

### Implementation

For each webhook type, ensure:

1. **WhatsApp webhook:** Use `channel_message_id` (Evolution API's external ID) in idempotency key
2. **Instagram webhook:** Use Meta's message ID in idempotency key
3. **Email webhook (if SendGrid):** Use SendGrid's message ID in idempotency key
4. **Realtime updates:** Use combination of entity ID + timestamp

### Webhook Logging

Store all webhook attempts (successful or not) in `webhook_logs`:

```sql
INSERT INTO webhook_logs (
  workspace_id, channel_type, event_type, payload, status, error, created_at
) VALUES (...);
```

**Use for debugging:** Can replay any logged webhook manually if needed.

## 3. Duplicate Event Protection

### Scenario

Multiple processes or webhook deliveries create duplicate conversation, message, or contact records.

### Protection Layer 1: Database Constraints

**Unique constraints:**
```sql
-- Contact deduplication per workspace
CREATE UNIQUE INDEX idx_contact_phone_per_ws 
  ON contacts(workspace_id, phone) WHERE phone IS NOT NULL;

CREATE UNIQUE INDEX idx_contact_email_per_ws 
  ON contacts(workspace_id, email) WHERE email IS NOT NULL;

-- Channel message uniqueness
CREATE UNIQUE INDEX idx_message_external_id 
  ON messages(workspace_id, channel_type, channel_message_id) 
  WHERE channel_message_id IS NOT NULL;
```

**ON CONFLICT handling:**
```sql
INSERT INTO contacts (workspace_id, phone, name, channel_origin)
VALUES (:workspace_id, :phone, :name, :channel_origin)
ON CONFLICT (workspace_id, phone) DO UPDATE SET updated_at = NOW()
RETURNING id;
```

### Protection Layer 2: Idempotency Keys

(See Section 1 — idempotency ensures no re-execution)

### Protection Layer 3: Application-Level Checks

Before any INSERT/UPDATE, verify:

```python
# Example: contact creation
existing_contact = db.query(Contact).filter(
  Contact.workspace_id == workspace_id,
  Contact.phone == phone
).first()

if existing_contact:
  return existing_contact  # Do NOT create duplicate
else:
  return Contact.create(...)
```

## 4. Poison Job Handling

### Definition

A background job that repeatedly fails and would retry infinitely, wasting resources.

**Examples:**
- AI API call returns consistent 400 error (bad prompt)
- Webhook delivery fails due to network partition (transient)
- Message send fails due to invalid phone number (permanent)

### Classification

| Type | Indicator | Action |
|---|---|---|
| **Transient** | Timeout, 502, 503 | Retry with exponential backoff |
| **Permanent** | 400, 401, 404, invalid data | Quarantine immediately |
| **Unknown** | Unexpected error | Quarantine after N retries |

### Quarantine Strategy

Create table:
```sql
CREATE TABLE dead_letter_queue (
  id UUID PRIMARY KEY,
  workspace_id UUID,
  job_type TEXT,  -- e.g., 'ai_suggestion', 'webhook_delivery'
  payload JSONB,
  error_message TEXT,
  error_code INT,
  attempt_count INT,
  last_attempted_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ,
  status TEXT  -- 'quarantined', 'manual_retry_pending', 'resolved'
);
```

### Quarantine Rules

**Permanent errors → Quarantine immediately:**
- Insert into `dead_letter_queue`
- Log error
- Alert admin (optional: Slack notification)
- Do NOT retry

**Transient errors → Retry with backoff:**
- Retry 1: +5 sec
- Retry 2: +30 sec
- Retry 3: +2 min
- Retry 4: +10 min
- Retry 5: +30 min
- After 5 retries: Quarantine

**Example:**
```javascript
// NestJS scheduled task
@Cron(CronExpression.EVERY_30_SECONDS)
async processBackgroundJobs() {
  const jobs = await this.jobQueue.find({ status: 'pending' });
  for (const job of jobs) {
    try {
      await this.executeJob(job);
    } catch (error) {
      if (this.isPermanentError(error)) {
        await this.quarantine(job, error);
      } else {
        job.retry_count++;
        job.next_retry = this.calculateBackoff(job.retry_count);
      }
    }
  }
}
```

### Manual Review

Admin endpoint: `GET /admin/dead-letter-queue`
- Shows quarantined jobs
- Manual retry capability
- Delete option (after investigation)

## 5. Retry Behavior

### Retry Strategy (Exponential Backoff + Jitter)

**For external API calls (Evolution, Anthropic, Groq, etc.):**

```javascript
const calculateBackoff = (attempt) => {
  const baseDelay = 1000; // 1 second
  const maxDelay = 60000; // 60 seconds
  const exponential = baseDelay * Math.pow(2, attempt);
  const jitter = Math.random() * exponential * 0.1; // ±10% jitter
  return Math.min(exponential + jitter, maxDelay);
};
```

**Retry limits:**
- Network calls (Evolution, Meta): max 5 retries (total ~3 min)
- AI calls (Anthropic, Groq): max 3 retries (total ~1 min)
- Database operations: max 2 retries (usually succeeds immediately)

### Circuit Breaker Pattern

**For frequently failing external services:**

```javascript
class CircuitBreaker {
  constructor(threshold = 5, timeout = 60000) {
    this.failureCount = 0;
    this.threshold = threshold;
    this.timeout = timeout;
    this.state = 'CLOSED'; // 'CLOSED' | 'OPEN' | 'HALF_OPEN'
  }

  async call(fn) {
    if (this.state === 'OPEN') {
      if (Date.now() - this.lastFailure > this.timeout) {
        this.state = 'HALF_OPEN';
      } else {
        throw new Error('Circuit breaker is OPEN');
      }
    }

    try {
      const result = await fn();
      this.failureCount = 0;
      this.state = 'CLOSED';
      return result;
    } catch (error) {
      this.failureCount++;
      this.lastFailure = Date.now();
      if (this.failureCount >= this.threshold) {
        this.state = 'OPEN';
      }
      throw error;
    }
  }
}
```

### What NOT to Retry

**DO NOT retry:**
- Duplicate request (idempotency key exists)
- Authentication failures (401, invalid token)
- Permanent data errors (400)
- Permission errors (403)

## 6. Dead Letter Queue & Manual Recovery

### Dead Letter Storage

Every failed job after max retries goes here:

```sql
INSERT INTO dead_letter_queue (
  workspace_id, job_type, payload, error_message, error_code, 
  attempt_count, created_at, status
) VALUES (...);
```

### Monitoring

Scheduled alert: if any job in DLQ for > 24 hours, notify admin.

### Manual Recovery

Admin can:
1. Review job details in DLQ
2. Investigate error
3. Fix underlying issue (e.g., add missing field)
4. Manually retry

```javascript
// Endpoint
POST /admin/dead-letter-queue/:job_id/retry
```

## 7. Queue Backpressure & Flow Control

### Scenario

Too many jobs queued; processing falls behind; memory pressure increases.

### Strategy

**Option 1: In-Memory Queue (NestJS on single dyno)**

- Queue size limit: 10,000 jobs in memory
- Incoming jobs when queue is full: drop and log error
- Alert: if queue size > 5,000, notify admin

**Option 2: External Queue (Future — not v1)**

- If queue grows beyond NestJS capacity, migrate to Redis or RabbitMQ
- v1: document this as scaling decision point

### Backpressure Handling

```javascript
async enqueueJob(job) {
  if (this.queue.length > 10000) {
    logger.warn('Queue backpressure: rejecting job', { job });
    throw new Error('Queue full; please retry');
  }
  this.queue.push(job);
}
```

**For webhook deliveries:**
- If backpressure detected, return 503 to caller
- Caller should retry (Evolution API will)
- Idempotency key prevents duplication

## 8. Worker Crash Recovery

### Scenario

NestJS dyno crashes; scheduled jobs (snooze timers, SLA checks, token refresh) are interrupted.

### Strategy

**NestJS Scheduled Tasks Persistence (v1 approach):**

1. Store last successful run in `cron_job_runs` table:

```sql
CREATE TABLE cron_job_runs (
  id UUID PRIMARY KEY,
  job_name TEXT UNIQUE,
  last_run_at TIMESTAMPTZ,
  status TEXT,  -- 'success' | 'error'
  error_message TEXT,
  updated_at TIMESTAMPTZ
);
```

2. On dyno restart, scheduled job checks:

```javascript
@Cron(CronExpression.EVERY_5_MINUTES)
async checkSLA() {
  const lastRun = await this.db.query(CronJobRuns).where({ job_name: 'sla_check' }).first();
  
  // If last run was > 10 minutes ago, we missed a run; catch up
  if (Date.now() - lastRun.last_run_at > 600000) {
    logger.warn('Catching up on missed SLA check');
    // Run with extra care; may have backlog
  }
  
  // Execute job
  await this.slaCheck();
  
  // Update last run
  await this.db.query(CronJobRuns).update({ last_run_at: Date.now() });
}
```

3. For **snooze timer** specifically:

```sql
-- Find all snoozed conversations whose snooze_until has passed
SELECT id FROM conversations 
WHERE status = 'snoozed' AND snooze_until < NOW()
-- Reopen them on next cron run (even if delayed)
```

**Idempotency ensures:** Reopening same conversation twice is safe (only changes status once).

### Monitoring

Alert if:
- Any scheduled job hasn't run in 30 min (should run every 5–10 min)
- Snooze queue has > 100 expired timers waiting for processing

## 9. Transactional Consistency

### Multi-Step Operations

Some operations span multiple database changes. All must succeed or all must fail.

**Example: Inbound message processing**

```javascript
// Pseudocode
async handleInboundMessage(payload) {
  await db.transaction(async (trx) => {
    1. Find or create contact  (idempotent)
    2. Find or create conversation
    3. Insert message record
    4. Update conversation.last_message_at
    5. Insert audit log
  });
  
  // After transaction succeeds:
  6. Broadcast realtime (outside transaction; async)
  7. Trigger AI jobs (outside transaction; async)
}
```

**If transaction fails:**
- Entire operation rolls back
- Caller receives error
- Can retry (idempotency key ensures no duplicate)

### Transaction Isolation Level

Use **READ_COMMITTED** (default for Postgres).
This prevents most phantom reads while maintaining performance.

## 10. Webhook Signature Validation

### Purpose

Verify that webhooks come from trusted sources (Evolution API, Meta, SendGrid, etc.).

### Implementation per Channel

**WhatsApp (Evolution API v2):**
```
Header: apikey = [instance_key]
Action: Compare with stored instance_key
```

**Instagram (Meta):**
```
Header: X-Hub-Signature-256 = sha256_hash
Action: Verify hash matches HMAC(payload, app_secret)
```

**Email (SendGrid):**
```
Header: X-Twilio-Email-Event-Signature = signature
Action: Verify signature
```

### Failure Handling

- Invalid signature → 401 Unauthorized; log attempt
- Missing signature → 401 Unauthorized; log attempt
- Valid signature but processing fails → 500 (retry later)

## 13. Failure Mode Summary

| Failure Mode | Detection | Mitigation | Recovery |
|---|---|---|---|
| **Duplicate webhook** | Idempotency key | Store result; return cached | Auto-idempotent |
| **Network timeout** | Timeout exception | Retry with backoff | Exponential backoff |
| **Permanent error** | HTTP 4xx | Quarantine job | Manual review in DLQ |
| **Poison job** | Max retries exceeded | Move to DLQ | Admin manual retry |
| **Dyno crash** | Cron job misses run | Catch-up on next run | Idempotent rerun |
| **Queue overflow** | Size > 10k | Reject with 503 | Caller retries |
| **Data corruption** | Constraint violation | Rollback transaction | Retry operation |
| **AI cost explosion** | Tokens logged | Track in `ai_usage_logs` | Plan gating enforces limit |
